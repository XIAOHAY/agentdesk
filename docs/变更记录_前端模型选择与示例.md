# 变更记录：前端 — Chat 模型选择器 + 主区示例问题

> 日期：2026-06-19
> 文件：`streamlit_app.py`　|　范围：仅 UI，不改后端逻辑

---

## 1. 动机

- 7B 模型会把数字写崩（详见《知识库统计修复》与 rerank 文档的 root cause），
  需要能**在界面直接换更强的模型**做对比、定位"是模型还是管线"。
- 面试官来体验时，应能**点示例直接跑**，不用打字。

## 2. 之前

- 模型只能在 `.env` 里配 `CHAT_MODEL`，**运行时不可切换**。
- 示例问题放在侧边栏"试一试"；而**侧边栏默认/被收起时根本看不到**（用户实测：找不到换模型与示例）。

## 3. 现在为什么这么改

1. **模型选择器放主区**（输入框正上方）：预设 `Qwen2.5-7B/14B/32B/72B`、`DeepSeek-V3` + "自定义…"。
   运行时覆盖 `settings.chat_model` 与 `os.environ["CHAT_MODEL"]`。
   因 `app/llm.py:chat()` 每次都**现读** `settings.chat_model`，**一个开关同时作用于
   改写(planner) / 生成(writer) / 裁判(critic) 三处**，无需改图或穿参。离线 fallback 时控件禁用并提示。
2. **示例问题"点击即问"搬到主区**输入框上方，5 个按钮各演示一个能力点：
   多文档 RAG / 计算器工具 / 知识库统计 / 套餐 SLA(近似重复检索) / 报销政策。
   置于**输入行之前**：按钮在本次 rerun 先于下方 `pop("pending_q")` 执行，故点击当次即填入并运行，
   **无需 `st.rerun()`**（规避旧版本 API 风险）；按钮 key 用 `demo_*`，与侧边栏 `s_*` 不冲突。

## 4. 权衡（为什么选 A）

| 方案 | 做法 | 代价 |
|---|---|---|
| **A 运行时覆盖 settings 单例（已采用）** | 选框 → 改 `settings.chat_model`+`os.environ` | 全局可变状态；但 Streamlit 每会话独立进程，可控 |
| B 穿参 `run_query`→nodes→`chat` | 无全局态、更"干净" | 要改 4 个函数签名 + state，改动面大，超出"加个开关"范围 |

## 5. 验证

- `app.llm.settings is settings` → **True**：改 `settings.chat_model` 后 `chat()` 立即用新模型（三处共用）。
- Read 逐段核对：模型块与示例块语法闭合、上下文顺接、无重复控件；侧边栏旧版模型块已移除（避免重复 widget）。
- 注：bash 沙箱对该文件挂载视图有同步延迟，故未用其做编译校验；真实文件以文件工具/Windows 路径为准，已 Read 核对。

## 6. 风险与回退

- **风险低**：纯前端展示层；后端 graph/检索/工具逻辑未动。
- **回退**：删除主区模型块与示例块即恢复；模型仍可由 `.env` 的 `CHAT_MODEL` 控制。

## 7. 使用

重新 `streamlit run streamlit_app.py`（或右上 Rerun）→ 输入框上方选 `Qwen2.5-32B+` → 点示例或直接提问。
仅在线模式（`.env` 配了 key）下换模型有效。
