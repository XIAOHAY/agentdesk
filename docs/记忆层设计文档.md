# AgentDesk · 记忆层（Memory Layer）技术设计文档

> 版本 v1.0 ｜ 状态：设计稿（待评审，**本文档不含落地代码**）
> 定位：在现有 `LangGraph 编排 + Qdrant 向量库 + Redis 缓存` 之上，补齐 Agent 的**分层记忆**能力。
> 对位 JD：「Memory 质量与演化」「Context 体系」「Memory 的生产、消费与持续进化链路」。

---

## 0. 结论先行（TL;DR）

AgentDesk 当前是一个**无状态**的单轮 Agentic RAG：每次 `/chat` 都从零检索、回答、丢弃。它缺的不是「骨架」，而是**跨轮、跨会话的记忆能力**。本设计在不破坏现有降级风格（无 key / 无 Qdrant 自动回退内存）的前提下，新增一个 `app/memory/` 子包，分三层：

| 层 | 解决什么 | 落点 | 存储 |
|---|---|---|---|
| 短期记忆 Working Memory | 多轮对话上下文 + 滚动 summary 压缩 | `app/memory/short_term.py` | Redis（会话态）/ 内存回退 |
| 长期记忆 Long-term Memory | 用户偏好/事实抽取→向量化→检索注入 | `app/memory/long_term.py` | **Qdrant**（按 user namespace 隔离）/ 内存回退 |
| 记忆演化 Memory Evolution | 写入去重、冲突更新、过期淘汰 | `app/memory/evolution.py` | 复用长期记忆存储 + 审计痕迹 |

编排侧在现有 `planner → retrieval → tool → writer → critic` 链路上，**入口加 `memory_retrieve`、出口加 `memory_write`、并在短期上下文超阈值时触发 `summarize`**。评测侧新增 `memory hit@k` 指标，复用现有 `eval/` 体系——**一鱼两吃**。

**Trade-off 一句话**：选择「复用 Qdrant 做长期记忆」放弃了「另起一个专用记忆库（如带图结构的记忆图谱）」的表达力，换来零新增基础设施、与现有 `qdrant_store` 接口完全一致、演示成本最低。这在一人项目阶段是正确的取舍。

## 1. 需求约束摘要

### 1.1 现状（已读代码确认）

- 编排：`app/graph/build_graph.py` 用 LangGraph 组装 `planner→retrieval→tool→writer→critic`，并带 `should_retry` 条件边；LangGraph 不可用时退化为 `_run_sequential` 顺序执行。
- 状态：`app/graph/state.py` 的 `AgentState(TypedDict, total=False)`，字段含 `query/plan/queries/evidence/tool_results/answer/citations/verify/iterations/trace`。**无任何记忆/会话字段**。
- 向量库：`app/rag/qdrant_store.py` 的 `QdrantStore`，接口 `add/search/save/load/chunks/__len__`；`store_factory.get_store()` 按 `settings.vector_backend` 选 Qdrant，失败回退 `VectorStore`（内存）。
- 缓存：`app/rag/cache.py` 的 `EmbeddingCache`，优先 Redis（`settings.redis_url`），连不上回退进程内 dict。
- 嵌入/对话：`app/llm.py` 暴露 `embed_texts(List[str])->List[List[float]]`（带缓存）、`embed_query(str)`、`chat(system, user)->str`；无 `OPENAI_API_KEY` 时走确定性 fallback（256 维哈希向量 + 模板拼接）。
- 入口：`app/api/main.py` 的 `/chat` 接收 `{"query": str}`，调用 `run_query(query)`，**请求体里没有 user_id / session_id**。

### 1.2 缺口

1. **无短期记忆**：多轮对话无法引用上文（"它" "刚才那个" 全部失效）。
2. **无长期记忆**：用户说过的偏好/事实（"我只看 2024 年的数据""我是法务"）下次会话全忘。
3. **无记忆演化**：即使存了记忆，也会越积越多、自相矛盾、永不过期。

### 1.3 非功能性约束（继承现有项目纪律）

- **零强依赖**：无 key / 无 Qdrant / 无 Redis 时必须仍能端到端跑通（走内存回退）。这是现有项目的核心卖点，记忆层不能破坏。
- **接口对齐**：长期记忆存储必须复用 `QdrantStore` 的接口风格，便于替换/测试。
- **可观测**：所有记忆动作要写入 `AgentState.trace`，与现有 trace 时间线一致，前端可视化。
- **成本意识**：summary 压缩与记忆抽取都要调 LLM，需有触发阈值，不能每轮都调。

## 2. 总体架构

### 2.1 数据流（在现有链路上的增量）

```
                 ┌─────────────────────────────────────────────┐
 /chat           │            LangGraph 编排（增量节点加粗）        │
 {query,         │                                              │
  user_id,  ───► │  memory_retrieve ─► planner ─► retrieval ─►   │
  session_id}    │       │                                tool   │
                 │       │ 注入: 短期 summary+近K轮             │   │
                 │       │       长期 召回记忆                  ▼   │
                 │       │                              writer      │
                 │       │                                ▼         │
                 │       │                              critic      │
                 │       │                                │         │
                 │       └──────────────────────────────►memory_write
                 │              (会话结束/每轮异步抽取)        │      │
                 └──────────────────────────────────────────┼──────┘
                                                             ▼
                          summarize（短期 buffer 超阈值时触发，压缩旧轮次）
                                                             │
   存储层:  Redis(会话 buffer + summary)   Qdrant(长期记忆, ns=mem:{user_id})
            └─ 连不上回退内存                └─ 连不上回退内存 VectorStore
```

### 2.2 模块划分

```
app/memory/
├── __init__.py
├── schema.py        # MemoryRecord / WorkingMemory 数据模型（dataclass）
├── short_term.py    # WorkingMemory: 对话 buffer + 滚动 summary
├── long_term.py     # LongTermMemory: 抽取→向量化→Qdrant 存取
├── evolution.py     # 写入去重 / 冲突更新 / 过期淘汰
└── store.py         # MemoryStore: 复用 QdrantStore 风格的记忆专用存储 + 内存回退
```

> 说明：记忆的向量存储**不与 RAG 知识库共用同一个 Qdrant collection**，而是独立 collection（见 §5.1），避免"知识"与"记忆"混在一起被检索。

## 3. 数据模型

### 3.1 长期记忆记录 `MemoryRecord`

```python
# app/memory/schema.py（设计签名，非实现）
@dataclass
class MemoryRecord:
    mem_id: str          # 稳定主键：md5(user_id + normalized_text) 便于去重定位
    user_id: str         # 归属用户（namespace 隔离用）
    text: str            # 记忆内容，如 "用户偏好只查询 2024 年财报"
    kind: str            # "preference" | "fact" | "event"（语义/情景记忆区分）
    embedding: List[float]
    created_at: float     # 首次写入时间戳
    updated_at: float     # 最近更新时间戳（冲突更新时刷新）
    last_used_at: float   # 最近被召回时间（LRU 淘汰依据）
    use_count: int        # 被召回次数（热度，淘汰时保护高频记忆）
    version: int          # 冲突更新自增，留可审计痕迹
    superseded_by: str | None  # 被哪条新记忆覆盖（审计链）
```

**字段设计理由**：
- `kind` 区分语义记忆（fact/preference，长期稳定）与情景记忆（event，可较快过期），让淘汰策略可按类型差异化。
- `version + superseded_by` 是"冲突更新留审计痕迹"的落点——不物理删旧值，而是标记其被新版本取代，对应 JD「Memory 的持续进化链路」可讲的点。
- `last_used_at + use_count` 支撑 LRU/LFU 混合淘汰。

### 3.2 短期记忆 `WorkingMemory`

```python
@dataclass
class WorkingMemory:
    session_id: str
    messages: List[dict]      # [{"role": "user"/"assistant", "content": str, "ts": float}]
    running_summary: str      # 旧轮次被压缩后的滚动摘要
    round_count: int          # 已发生的对话轮数
```

### 3.3 AgentState 扩展（对 `app/graph/state.py` 的增量）

在现有 `AgentState` 上新增字段（保持 `total=False`，向后兼容；旧调用方不传则为空）：

```python
class AgentState(TypedDict, total=False):
    # —— 现有字段保持不变 ——
    query: str
    ...
    trace: List[dict]
    # —— 记忆层新增 ——
    user_id: str                       # 记忆归属
    session_id: str                    # 会话标识
    working_memory: dict               # WorkingMemory 序列化（短期上下文）
    recalled_memories: List[dict]      # 本轮召回的长期记忆（注入 prompt 用）
    memory_writes: List[dict]          # 本轮抽取/写入的记忆（trace 与审计用）
```

## 4. 三层设计详述

### 4.1 短期记忆 Working Memory（`short_term.py`）

**目标**：让多轮对话有上下文，又不让 prompt 无限膨胀。

**策略**：滑动窗口 + 滚动摘要。保留「`running_summary` + 最近 K 轮原文」；当 `round_count` 超过 N 触发 `summarize`，把"旧轮次 + 旧 summary"用一次 LLM 压成新 `running_summary`，只保留最近 K 轮原文。

**关键参数**（进 `config.py`，见 §6）：`mem_short_window_k`（保留原文轮数，默认 4）、`mem_summarize_every_n`（触发摘要的轮数阈值，默认 8）。

**接口签名（设计）**：

```python
class WorkingMemory:
    def load(self, session_id: str) -> "WorkingMemory": ...      # 从 Redis/内存读
    def append(self, role: str, content: str) -> None: ...        # 追加一轮
    def need_summarize(self) -> bool: ...                         # round_count > N ?
    def build_context(self) -> str: ...                           # summary + 近K轮 → prompt 片段
    def persist(self) -> None: ...                                # 写回 Redis/内存
```

**存储**：复用 `cache.py` 的 Redis 连接思路——key=`wm:{session_id}`，value=JSON，`ex=86400`（会话级 TTL）；Redis 连不上回退进程内 dict。**不新增依赖**。

**摘要触发（`summarize` 节点）**伪流程：
```
if working_memory.need_summarize():
    old = 除最近K轮外的全部 messages + 旧 running_summary
    new_summary = chat(system="把对话压成要点，保留事实与未决问题", user=old)
    working_memory.running_summary = new_summary
    working_memory.messages = 最近K轮
```

> Trade-off：摘要会损失细节。缓解——摘要 system prompt 明确"保留已确认事实、用户偏好、未决问题"，并把会触发记忆抽取的内容同时沉淀到长期记忆（见 4.2），避免压缩丢信息。

### 4.2 长期记忆 Long-term Memory（`long_term.py`）

**目标**：跨会话记住用户偏好/事实，检索时按相关性召回并注入。

**生产（写入）**：从对话里抽取"用户偏好/事实"。两种抽取触发，二选一或叠加：
1. **规则触发**（零成本，先做）：命中模式词（"我是…""我只看…""以后都…""记住…"）即抽取。
2. **LLM 触发**（更全，可选）：每轮或会话结束让 LLM 判断"这轮有无值得长期记住的信息"，输出结构化 `{kind, text}` 列表。

抽取出的文本 → `embed_texts()` 向量化 → 经 `evolution` 去重/冲突处理 → 写入 `MemoryStore`。

**消费（检索）**：`memory_retrieve` 节点用当前 query 向量在该 user 的 namespace 内召回 Top-M 记忆，注入 writer 的 prompt（与现有 `evidence` 并列，但来源标注为 `[memory]`）。

**接口签名（设计）**：

```python
class LongTermMemory:
    def retrieve(self, user_id: str, query: str, top_k: int = 3) -> List[MemoryRecord]: ...
    def write(self, user_id: str, items: List[dict]) -> List[MemoryRecord]: ...
    #   items = [{"kind": "...", "text": "..."}]；内部走 evolution 去重/冲突
    def extract(self, query: str, answer: str) -> List[dict]: ...  # 规则/LLM 抽取
```

**注入 writer 的方式**（对 `nodes.py:writer_node` 的增量，非改写）：在现有 `context` 前追加一段：
```
【关于该用户已知信息】
- (preference) 用户偏好只查询 2024 年财报
- (fact) 用户身份：法务
```
并在 system prompt 补一句"结合【关于该用户已知信息】理解意图，但事实仍以【参考资料】为准"。这样既个性化又不破坏现有"答案必须有引用、不编造"的纪律。

### 4.3 记忆演化 Memory Evolution（`evolution.py`）— JD 最看重的差异点

三个机制，全部作用在"写入长期记忆"这一刻：

**① 写入去重**：新记忆向量与该 user 已有记忆做相似度检索，`max_sim > τ_dedup`（默认 0.92）→ 判定为同一条，**更新 `updated_at/use_count` 而非新增**。

**② 冲突更新**：相似度落在中间带（`τ_conflict < sim ≤ τ_dedup`，如 0.80~0.92）且 `kind=fact` → 视为"同一事实的新表述/新取值"，**用新值覆盖，旧值 `version+1`、置 `superseded_by=新 mem_id`，保留审计痕迹**（带时间戳）。
- 例：旧"用户在北京" → 新"用户搬到上海" → 覆盖，旧记录留痕可审计。

**③ 过期淘汰**：两条路并行——
- **TTL**：`event` 类记忆默认 30 天过期（`fact/preference` 不设 TTL）。
- **LRU/容量上限**：单 user 记忆数超 `mem_max_per_user`（默认 500）时，按 `last_used_at` 升序 + `use_count` 加权淘汰最冷的。

**接口签名（设计）**：

```python
class MemoryEvolution:
    def resolve_write(self, user_id: str, new: MemoryRecord) -> WriteDecision: ...
    #   WriteDecision ∈ {INSERT, UPDATE(existing_id), CONFLICT_OVERWRITE(old_id)}
    def evict_if_needed(self, user_id: str) -> List[str]: ...  # 返回被淘汰 mem_id
```

**阈值表（可配，进 config）**：

| 参数 | 默认 | 含义 |
|---|---|---|
| `mem_dedup_threshold` | 0.92 | ≥ 则去重（更新不新增） |
| `mem_conflict_threshold` | 0.80 | (此值, dedup) 区间且同类→冲突覆盖 |
| `mem_event_ttl_days` | 30 | event 类过期天数 |
| `mem_max_per_user` | 500 | 单用户记忆上限，超则 LRU 淘汰 |

> Trade-off：阈值法简单可控、零额外模型；代价是边界 case 可能误判（近义但不同的事实被当成冲突）。缓解——冲突覆盖只对 `kind=fact` 生效，且**留审计痕迹可回溯**，不做物理删除。后续若要更准，可升级为"LLM 判定是否同一事实"，接口不变。

## 5. 与现有模块的接入点（复用 Qdrant）

### 5.1 MemoryStore：复用 QdrantStore 风格（`store.py`）

新增独立 collection（不污染 RAG 知识库）：`settings.mem_collection`（默认 `agentdesk_memory`）。`MemoryStore` 对齐 `QdrantStore` 接口风格，并增加记忆专属的 namespace 过滤与 upsert：

```python
class MemoryStore:                       # 镜像 QdrantStore 的可选依赖+回退模式
    def upsert(self, rec: MemoryRecord) -> None: ...
    def search(self, user_id: str, vec: List[float], top_k: int)
              -> List[tuple[MemoryRecord, float]]: ...   # 带 user_id payload 过滤
    def delete(self, mem_ids: List[str]) -> None: ...
    def list_by_user(self, user_id: str) -> List[MemoryRecord]: ...
```

**Qdrant 实现要点**：
- payload 存 `user_id/kind/text/created_at/updated_at/last_used_at/use_count/version/superseded_by`。
- 检索时用 Qdrant `Filter(must=[FieldCondition(key="user_id", match=MatchValue(user_id))])` 做 namespace 隔离（同一 collection、payload 过滤实现多租户，运维最省）。
- `point id = uuid5(NAMESPACE_URL, mem_id)`，与现有 `qdrant_store.py` 完全一致的 id 生成方式 → 天然支持 upsert 覆盖。

**回退**：`store_factory` 增加 `get_memory_store()`，`vector_backend=qdrant` 且可连 → Qdrant；否则回退一个内存版 `MemoryStore`（dict + numpy 余弦，逻辑同 `store.VectorStore`）。**与现有 `get_store()` 完全同构**。

### 5.2 LangGraph 编排改造（`build_graph.py` / `nodes.py`）

新增 3 个节点，**只在首尾和条件处插入，不改动现有 5 节点内部逻辑**：

```python
g.add_node("memory_retrieve", memory_retrieve_node)   # 新
g.add_node("planner", planner_node)
g.add_node("retrieval", retrieval_node)
g.add_node("tool", tool_node)
g.add_node("writer", writer_node)
g.add_node("critic", critic_node)
g.add_node("memory_write", memory_write_node)          # 新
g.add_node("summarize", summarize_node)                # 新

g.set_entry_point("memory_retrieve")
g.add_edge("memory_retrieve", "planner")
# ...原有 planner→retrieval→tool→writer→critic 不变...
g.add_conditional_edges("critic", should_retry, {"retry": "retrieval", "end": "memory_write"})
g.add_conditional_edges("memory_write", need_summarize_edge, {"summarize": "summarize", "end": END})
g.add_edge("summarize", END)
```

`_run_sequential` 同步补齐（保持 LangGraph 不可用时等价）：开头 `memory_retrieve`、`should_retry=="end"` 后 `memory_write` →（必要时）`summarize`。

**节点职责**：
- `memory_retrieve_node`：load 短期 WorkingMemory + 召回长期记忆 → 写 `working_memory/recalled_memories` 到 state + trace。
- `memory_write_node`：抽取本轮记忆 → 经 evolution → upsert；append 本轮到 WorkingMemory 并 persist → 写 `memory_writes` 到 trace。
- `summarize_node`：见 §4.1，压缩短期 buffer。

### 5.3 API 改造（`api/main.py`）

`ChatRequest` 增加可选字段，向后兼容（不传则用默认匿名用户与新会话）：

```python
class ChatRequest(BaseModel):
    query: str
    user_id: str = "anonymous"
    session_id: str | None = None     # 不传则后端生成
```

`run_query(query)` 签名扩展为 `run_query(query, user_id="anonymous", session_id=None)`，旧调用方（如 `streamlit_app.py`、`eval/`）不传照常工作。`ChatResponse` 增加 `recalled_memories` 字段用于前端展示"本轮用到的记忆"。

## 7. 评测：memory hit@k（复用 `eval/`，一鱼两吃）

**指标定义**：构造"先植入记忆 → 后续提问应当用到该记忆"的样本对，衡量「应被想起的记忆有没有被召回到 Top-k」。

```
memory_hit@k = (被正确召回到 Top-k 的样本数) / 总样本数
```

**数据集**（仿现有 `eval/dataset.jsonl` 风格，新增 `eval/memory_dataset.jsonl`）：
```json
{"user_id":"u1","seed":["我只关注 2024 年的财报"],"query":"营收增长怎么样？","expect_mem":"2024 年财报"}
```

**流程**（新增 `eval/run_memory_eval.py`，复用 `LongTermMemory.write/retrieve`）：
1. 对每条样本，先 `write(user_id, seed)` 植入；
2. 用 `query` 调 `retrieve(user_id, query, top_k=k)`；
3. 判断 `expect_mem` 是否命中召回结果（文本包含或向量相似度阈值）；
4. 汇总 `hit@1 / hit@3 / hit@5`，输出到 `eval/reports/`（对齐现有报告格式）。

> 这条让"记忆层"直接接进你已有的评测强项，简历可写"为记忆层设计 memory hit@k 评测，hit@3 达 X%"。

## 8. 风险与盲区（单人作战视角）

1. **多轮一致性 vs 成本**：summarize 和 LLM 抽取都要调模型。盲区——若每轮都调，延迟/成本翻倍。对策：摘要走阈值触发（每 N 轮一次）、抽取先用规则触发，LLM 抽取设为可选开关 `mem_extract_mode`。
2. **隐私与多租户**：记忆按 `user_id` 隔离，但 payload 过滤是逻辑隔离非物理隔离。盲区——`user_id` 必须来自可信来源，不能由前端任意伪造（演示项目可接受，生产需鉴权）。文档需注明。
3. **记忆污染/越权写入**：恶意对话可能写入误导性"事实"。对策——抽取来源限定于"用户自述"，并保留审计痕迹可回滚（`superseded_by`/`version`）。
4. **冷启动**：新用户无记忆，`memory_retrieve` 返回空，链路必须优雅跳过（注入段为空字符串），不能报错。
5. **回退一致性**：内存版 MemoryStore 进程重启即丢，仅用于无 Qdrant 的演示；需在 README 明确"持久记忆需配 Qdrant"。

## 9. 演进路径

```
Phase 1（本设计，3-5 天）：三层基础 + 3 节点 + memory hit@k
   ├─ 短期：Redis buffer + 滚动 summary
   ├─ 长期：Qdrant namespace 存取 + 规则抽取
   └─ 演化：阈值去重/冲突覆盖/TTL+LRU
        ↓
Phase 2：抽取升级为 LLM 结构化抽取；冲突判定升级为 LLM 判同
        ↓
Phase 3：情景记忆（episodic）+ 时间衰减打分；记忆可视化面板（前端 trace 已有基础）
        ↓
Phase 4：记忆图谱（实体-关系），支持多跳"为什么记得这件事"
```

每个 Phase 接口不变，只换内部实现——这是本设计把"阈值/规则"与"接口"解耦的目的。
