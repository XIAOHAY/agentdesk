# 变更记录：Rerank 重排升级（含决策与实验留痕）

> 文件：`app/rag/rerank.py`　|　范围：仅 rerank（不动 indexer/store/retriever 接口）
> 目的：回答「前端示例问题有的好有的坏」的根因，并把 rerank 从「能跑」升级到「可解释、可上线」。

---

## 1. 背景：为什么会「有的好有的坏」

知识库 `data/docs/plan_AC-100 ~ AC-119` 是**近似重复文档**——19 篇概述段落一字不差，
只有「型号 / 地域 / 月费 / SLA / 存储 / 并发」几个数字不同。对这种数据：

- **带唯一 ID 的问题**（如「AC-110 的 SLA」）：BM25 关键词召回能精确命中型号 → 稳 → **好**。
- **反向 / 属性查询**（如「部署在西南、月费 4320 元的那个套餐叫什么型号？」）：
  问题里没有型号，多篇候选高度相似，**排序全靠 rerank** → 一旦 rerank 区分力不足就飘 → **坏**。

量化佐证（离线索引，eval/dataset.jsonl 37 条）：

| 配置 | hit@k | MRR |
|---|---|---|
| vector（纯向量） | 0.919 | 0.800 |
| hybrid（向量+BM25，无 rerank） | 1.000 | 0.941 |
| hybrid + rerank | 1.000 | **1.000** |

→ 召回（hit@k）早已 1.0，**真正决定好坏的是排序（MRR），而排序由 rerank 决定**。

---

## 2. 之前为什么那么做（旧实现：词项重叠占比）

```
score = |q ∩ d| / |q|      # 命中的 query 词数占比
```

- **动机**：零依赖、可离线、接口先跑通，契合本项目「无 API key 也能端到端跑」的定位。
- **它其实已经够用**：本评测集每条问题都含一个**逐字唯一 token**（型号 ID 或精确月费），
  gold 文档恰好多命中这一个词，overlap 就把它排到第 1，MRR 已达 1.0。
- **局限**：overlap 把「样板词」和「唯一区分词」**等权**——靠的是「多匹配一个词」的数量优势，
  而不是「这个词更有区分力」。一旦 query 没有逐字唯一 token（改写、口语化），
  或候选长度参差导致匹配数被样板词拉平，它就只能按噪声排序。

---

## 3. 踩过的坑：纯 IDF 加权（已验证退化，放弃）

**设想**：把打分整体换成「命中词的 IDF 之和」，让区分词主导、压低样板词。

**实测结果**：离线 hybrid+rerank 的 **MRR 从 0.941 掉到 0.595**（明显退化）。

**复盘原因**：中文 query 分词后会混入「是多少 / 可用 / 性是」等**疑问词、助词 bigram**，
它们在模板化的 plan 文档里几乎不出现 → IDF 很高；一旦偶然出现在某篇 off-topic 文档
（例如 `sample_company.md`），该文档分数被抬到 gold 之上。

实例（query=「AC-110 套餐的 SLA 可用性是多少？」）：

```
纯IDF top: sample_company.md(0.563) > plan_AC-110.md(0.449)   ← 正确文档被挤到第2
旧overlap top: plan_AC-110.md(0.60) > 其他(0.55)              ← 正确文档第1
```

**教训**：IDF 会放大「稀有但无关」的噪声词，**不能单独当主排序键**。

---

## 4. 现在为什么这么改（最终实现）

### 4.1 离线兜底 = overlap 主键 + IDF 同分裁决

```
score = 命中 query 词数(整数)  +  IDF 加成(∈[0,1))
```

- **整数部分 = overlap 排序**：与旧实现同序，**实证 MRR 1.0，保证不退化**。
- **小数部分 = IDF 加成**：归一化到 `[0,1)` 后封顶 `<1`，**绝不跨越整数边界** →
  只在「匹配词数相同」的并列文档之间，让命中更有区分力词（高 IDF）的排前面。
- 既保留 overlap 的稳健，又补上它「同分排不出」的短板；仍然**零依赖、可离线**，是默认路径。
- IDF 在**候选池内**统计（而非全库）：rerank 只需在召回回来的相似文档间做区分，
  候选池内的文档频率恰好放大彼此差异，自包含、免改外部接口。

### 4.2 可插拔 cross-encoder（生产级正解）

设环境变量 `RERANK_MODEL`（如 `BAAI/bge-reranker-v2-m3`）且依赖可用时，
自动改用 cross-encoder 对 (query, doc) 做语义相关性打分；否则回退到 4.1。

```bash
# 开启生产级 rerank（需自行 pip install sentence-transformers）
export RERANK_MODEL=BAAI/bge-reranker-v2-m3
```

- cross-encoder 能处理「改写 / 无逐字 token」的难例，是真正解决近似重复排序的方案。
- 默认**不**加载重模型，避免破坏「无 key/无依赖也能跑」与意外下载大模型。
- 一份 `rerank()` 接口：离线能跑、上线能升级，互不破坏。运行期异常自动落回离线兜底。

## 5. 验证（离线索引，apples-to-apples）

| 配置 | hit@k | MRR | 结论 |
|---|---|---|---|
| hybrid 无 rerank（基线） | 1.000 | 0.941 | rerank 的价值基准线 |
| 旧：纯 overlap | 1.000 | 1.000 | 本数据集已达上限 |
| 纯 IDF（废弃方案） | 1.000 | 0.595 | ❌ 退化，已放弃 |
| **新：overlap 主键 + IDF 裁决** | **1.000** | **1.000** | ✅ 不退化，且补强同分场景 |

> 结论：新实现**在现有评测集上与 overlap 持平（无退化）**，额外获得同分裁决能力与
> cross-encoder 升级路径。真正的指标跃升需要更难的评测集（改写/反向/无逐字 token 的难例）
> 与 cross-encoder，这是下一步。

---

## 6. 风险与回退

- **风险低**：默认路径排序与旧 overlap 同序；cross-encoder 默认关闭。
- **回退**：删除 `RERANK_MODEL` 即回到离线兜底；若需完全回到旧行为，
  把 `rerank()` 离线分支换回 `_overlap_score_legacy`（已保留该函数作对照）。

> 因为 IDF 放大了疑问词等噪声 token。于是改成 **overlap 主键 + IDF 仅做同分裁决**，
