# 变更记录：知识库统计 / 工具类回答可信度修复

> 日期：2026-06-19
> 文件：`app/graph/judge.py`、`app/graph/nodes.py`、`app/tools/builtins.py`
> 范围：critic 验证层 + writer 提示 + kb_stats 输出　|　向后兼容、可回退

---

## 1. 现象

"知识库里有多少个文档？"这类**统计/计数**问题经常答错或答案不稳定，而且**换大模型也一样**——
说明根因不在生成模型能力，而在管线与验证层。

## 2. 根因（均与模型大小无关）

1. **验证层对工具失明（核心）**：`critic_node` 调 `judge()` 时只传了检索证据、**没传 `tool_results`**。
   faithfulness 只按"答案能否被检索证据支撑"打分；而"24 个文档"这个数字来自 **kb_stats 工具**，
   检索回来的 5 个 chunk 里根本没有 → 正确答案被判"证据支撑不足"→ 触发 critic 重试最多 3 轮 → 答案飘。
   即便是 72B，也照样被这个**验证逻辑缺陷**否决。
2. **检索喧宾夺主**：每条查询都跑检索，注入 5 个不相关 chunk，界面还显示"命中证据 5"，
   模型容易被"5"带偏或去数 chunk_id。
3. **工具输出有歧义 + 诱导重数**：旧 `kb_stats` 返回 `chunks=X, docs=Y: <全部文件名>`——
   两个数字（chunk/doc 概念易混），又把整列表丢给模型让它自己数（LLM 数长列表本就易错）。

## 3. 之前为什么那么做

- **judge 只看检索证据**：初版把 RAG 的 faithfulness 等同于"答案 vs 检索片段"（RAGAS 标准思路），
  零依赖、对纯检索问答够用；但忽略了**工具结果同样是事实依据**。
- **kb_stats 返回完整列表**：为了可观测/透明，方便 demo 直接看到库里有哪些文档。

## 4. 现在为什么这么改

1. **`judge()` 增加可选参 `tool_results`**：LLM 裁判把工具输出并入证据上下文，并在 system 中声明
   "工具结果（如统计数字）视为权威依据"；启发式也把工具 token 计入支撑集。
   **缺省（None）时行为与改动前完全一致 → 向后兼容**。
2. **`critic_node` 传入 `state["tool_results"]`**，让验证层"看得见"工具算出的事实。
3. **writer 系统提示加一句**："涉及计数/统计的数字，以【工具结果】给出的为准、直接采用，不要自行数文档或列表。"
4. **`kb_stats` 输出**把"文档总数"标为**权威值**，保留 chunk 数与列表但弱化重数诱导。

## 5. 验证

| 用例 | 结果 | 说明 |
|---|---|---|
| judge 不传 / 传空 `tool_results` | score 0.0（与旧一致） | ✅ 向后兼容，其余查询零影响 |
| LLM 裁判路径（mock chat） | method=llm, **faithful=True, score 1.0** | ✅ 工具结果进入裁判上下文，"24 个文档"判为可信 |
| 启发式兜底 | 0 → 0.25 | 方向正确，但超短数字答案仍低于 0.6 阈值（token 重叠粗启发式的固有局限） |

> 完整鲁棒性依赖一个**能稳定吐 JSON 的裁判模型**（弱模型解析失败会回退到启发式）——
> 这与"前端可切换更强模型"联动。

## 6. 风险与回退

- **风险低**：只影响"有工具结果"的查询（kb_stats / calculator），纯检索问答零影响。
- **回退**：`judge()` 调用去掉第 4 个实参即恢复旧行为；writer 提示、kb_stats 输出各自独立可还原。

## 7. 未做（谨慎留项）

- **路由仍是硬关键词**：`["多少篇","多少个文档","知识库","文档数量"]`。换问法（"几份资料""库里几个文件"）
  可能不触发工具。扩词有**误触发**风险（含"文档"的内容类问题会被误判为统计），故本次不动，列为后续。

## 8. 面试一句话

> "kb_stats 算的数本来是对的，但我的 critic 做 faithfulness 时只看检索证据、没纳入工具结果，
> 导致工具类正确答案被判不可信并反复重试——这是**验证层缺陷，与模型大小无关**。
> 我让 critic 感知工具结果、把统计数字声明为权威依据，并让 writer 直接采用工具数字不重数。
> 改动向后兼容，单测验证：不传工具结果行为不变，传入后 LLM 裁判把'24 个文档'正确判为可信。"
